1. Testing with PractRand
***********************************************************************

1.A - Connecting your RNG to PractRand
If you're testing an RNG that is not built in to PractRand then your 
first challenge is getting the data to PractRand in a manner that it 
can test.  The three general ways to interface an external RNG to a 
test are by piping random data in (aka stdin), by dumping random data 
to a file, or by linking the RNG to the tester.  

1.A.1 - Testing a program's output:
If you have a program that generates raw random data and sends them to 
standard output then you can pipe them in to the PractRand RNG testing 
tool and it can work with them.  If your program is called My_RNG then 
the command line looks like this on linux:
My_RNG | ./RNG_test stdin
Or like this on Windows (from a command prompt):
My_RNG | RNG_test stdin

If your program stops sending out random data before the PractRand RNG 
testing tool is done testing then it may print "error reading input".  
In that case you might want to try adding command line options like 
"-tlmin 1KB", which will tell RNG_test that you want it to start 
printing reports ASAP - by default RNG_test does a few seconds of 
testing before printing any output, regardless of how fast the testing 
goes.  RNG_test will still print "error reading input" when the input 
stops, but at least you'll get some test results as long as the input is 
at least 1024 bytes long.  

If you know that your program produces data in 8 bit chunks then you 
should change "stdin" to "stdin8".  Similar variants exist for 16, 32, 
and 64.  If you don't know the format of your data or none of those 
make any more sense than the others then just use "stdin".  

1.A.2 - Testing a file:
If you have a file of random data named My_Data that you want to test, 
technically PractRand doesn't support that.  But it's easily adapted 
to piped in data using standard tools on most OSes.  
On linux, try:
cat My_Data | ./RNG_test stdin
On windows instead use:
type My_Data | RNG_test stdin

The comments that apply to testing data from a program apply here as 
well.  In fact the comment on "error reading input" becomes much more 
important.  

Note that reading random data from a file tends to be much slower 
than getting it by any other method.  

1.A.3 - Testing your RNG at full speed:
It's also possible to link your RNG directly with PractRand and/or 
RNG_test.  This eliminates the overhead of piping the data around, 
increasing performance for fast RNGs.  However, it tends to require 
more work as you have to understand some of the subtleties of the 
internals of PractRand.  The easiest way may be to to edit the 
"dummy" RNG in dummy_rng.h in the PractRand tools directory, but 
other ways are possible too.  

1.A.4 - Options NOT supported by PractRand
PractRand supports ONLY packed binary data as input.  You can not 
give it ASCII 0s and 1s or ASCII hex or floating point data or 
anything else that doesn't have a full 8 random bits in each and 
every byte.  

1.B - Interpretting Results
The next challenge is figuring out what the results mean.  

1.B.1 - RNG_test results for people in a hurry
If you are using RNG_test the simplest way to interpret test 
results is to look for the word "FAIL" in the output.  It 
will appear on the right-hand side and be obvious, like this.  
Here is an example results summary with 3 such evaluations:

rng=sfc_v2_16, seed=0x7a15cb31
length= 32 kilobytes (2^25 bytes), time= 4.7 seconds
  Test Name                     Raw         Processed     Evaluation
  BCFN(2+0,13-0)                R=  -6.5    p =1-3.5e-3   unusual
  [Low4/16]BCFN(2+0,13-0)       R= +39.2    p =  4.4e-20    FAIL !!!
  [Low4/16]BCFN(2+1,13-1)       R= +28.8    p =  3.1e-14    FAIL !!
  [Low1/16]BCFN(2+0,13-1)       R= +27.1    p =  2.1e-13    FAIL !!
  ...and 50 test result(s) without anomalies

1.B.3 - RNG_test results in more detail
RNG_test produces a series of interim result summaries as it goes 
along.  Each result summary has a header showing the RNG name, the 
number of bytes tested, the time taken, and the RNG seed used (so 
you can reproduce the results later if desired).  The body of the 
result summary is a table showing all anomolous results followed 
by a statement of how many results were omitted from the table 
because they were not anomolous.  If the table would have zero 
entries then the table is skipped.  If no results were omitted then 
the number of omitted results is skipped.  The threshold for what 
kinds of results are considered anomolous for this purpose is very 
adjustable via command line parameters.  Adding "-a" will force all 
results to be shown in the table regardless of how suspicious they 
were or weren't.  

The table has four columns:
"Test Name" - a name for the subtest the line corresponds to
"Raw" - not of much use to end users, but I use this a lot
"Processed" - either a p-value or "pass" or "fail".  However, 
see 3.B.3 for more details on RNG_test p-values
"Evaluation" - an english word or phrase describing the result.  
"FAIL" means that the RNG unambiguously failed that subtest, 
while "normal" means it unambiguously passed.  Particularly 
extreme results (like "FAIL") are indented two extra spaces to 
make spotting them easier when scrolling past large numbers of 
results.  The complete list of possible evaluations, in order 
from least suspicious to most, is: "normal", "normalish", 
"unusual", "mildly suspicious", "suspicious", "very suspicious", 
"VERY SUSPICIOUS", "FAIL".  In addition, "FAIL" messages may 
have varying numbers of exclamation marks printed after them - 
the more egregious the failure, the more exclamation marks.  

1.B.4 - Notes on evaluations
The "Evaluation" column assigns results based upon dynamic p-value 
thresholds.  The false positive rate for "FAIL" messages is about 0.000000003 
per results summary.  This means that if you see a billion different results 
summaries about good PRNGs, the total number of "FAIL" messages is expected to 
be 3.  The false positive rate for "unusual" messages is 0.1 per results 
summary - if ten results summaries are printed there are expected be 1.0 
"unusual"s.  These are averages.  If multiple "unusual"s are produced at once, 
don't read too much in to it - there is some small degree of correlation 
between different test results, so two or three "unusual"s is less suspicious 
than a single "mildly suspicious".  My usual recommendation is to focus on the 
single worst evaluation in any single results summary.  

Note that the false positive rate being specified on a per-results-summary 
basis is different from other test suites, which often specify false positive 
rates on a per-p-value basis.  This is done because the number of p-values 
produced, and their relative importance, may vary wildly, and be difficult to 
fully take in to account even for those with extensive knowledge of PractRand, 
almost impossible for ordinary users.  Thus I endeavor to tell users in plain 
English which results are failure, which are suspicious, and which are 
perfectly normal.  However, this is done in the context of a single results 
summary - if you run RNG_test with the right command line parameters it might 
print 30+ results summaries for various different test lengths, in which case 
it would be perfectly normal for a few "unusual" messages to appear.  

For a concrete example: if you run RNG_test with the command line parameters 
"RNG_test hc256 -tlmin 30 -tlmax 35" then it will test a functionally perfect 
PRNG and print 6 different results summaries - one each for 1 gigabyte, 2 
gigabytes, 4 gigabytes, 8 gigabytes, 16 gigabytes, and 32 gigabytes.  Thus the 
expected number of "FAIL" messages for the entire run would be about 
0.000000003 * 6 = 0.000000018, while the expected number of "unusual" messages 
would be about 0.1 * 6 = 0.6.  

1.B.5 - Notes on RNG_test p-values
When RNG_test prints a p-value it can be in any of the 
following forms:
* normal p-value (e.g. "p = 0.188") - exactly what you'd expect
* p-value near zero (e.g. "p = 2.3e-4") - scientific notation in 
a fairly standard form.  Note however that the exponent can reach 
extreme values (up to 9999) that do not fit in normal floating 
point formats.  Also note that the number of significant digits 
tends to be small.  
* p-value near one (e.g. "p =1-2.7e-3") - one minus a number in 
scientific notation.  This format is a little less standard but 
still fairly self-explanatory and it allows finer resolution to 
be displayed for numbers close to one.  Again, the exponent can 
exceed those normally allowed in floating point numbers, up to 
9999.  That also means that despite the low number of significant 
digits, the resolution of values can exceed that normally found 
in floating point numbers near 1.  
* p-values equal to zero (e.g. "p = 0") - values VERY close to 
zero may get rounded to zero, if the exponent would exceed 9999.  
* p-values equal to one (e.g. "p = 1") - values VERY close to one 
may get rounded to one, if the exponent would exceed 9999.  
* p-values that are only very crudely approximated - these can be 
distinguished from normal p-values by the "~" between the "p" and 
the "=" in the Processed results column of RNG_test output.  Such 
results may be shown with fewer significant digits displayed than 
normal p-values.  
* "pass" or "fail" in place of a p-value - Some subtests can't 
produce p-values.  If you must have a p-value, assume that "pass" 
corresponds to a p-value randomly distributed between 0 and 1, 
while "fail" corresponds to a p-value of exactly zero.  Generally 
PractRand subtests that produce this kind of output are very 
boring tests that never fail any but the most absurdly extreme 
RNGs (things like RNGs that produce only zeroes).  These results 
are necessary because some normal subtests results cannot be 
calculated at all under such absurd circumstances, but the test 
still needs some way to report those circumstances occuring.  

An example results chart showing one of each category of result:
  Test Name                     Raw         Processed     Evaluation
  BRank(12):512(2)              R=  +0.4    p~= 0.4       normal
  DC6-9x1Bytes-1                R=  +0.4    p =1-1.1e-3   unusual
  Gap-16:!                      R=  +0.0      "pass"      normal
  Gap-16:A                      R=  +0.5    p = 0.511     normal
  Gap-16:B                      R=  +3.3    p =  9.9e-3   normal

1.B.6 - TestResult objects from calling PractRand directly
PractRand tests natively produce sets of results of type 
PractRand::TestResult.  Each of these objects correspond to a subtest 
result.  The simplest thing to do with them is call their get_pvalue() 
method.  See tests.h for details on their interface, or RNG_test.cpp for 
examples of more complicated things you can do with them.  

1.B.7 - Miscellaneous PractRand test result quality notes:
The various tests within PractRand vary widely in terms of the quality of 
p-values they produce - the tests used at default settings (the "core" tests) 
tend to produce cleaner p-values than other tests in PractRand, because I 
spent more effort on them.  In all cases however I have made effort to 
minimize any chance of the tests themselves contributing to a false positive - 
when there is uncertainty what the true p-value result should be, I attempt to 
report the least extreme possibility within the range of uncertainty.  

Very short tests often have oddities in the p-value quality, especially for 
results near 1.  Some tests are actually incapable of produce a result close 
enough to 1.0 to qualify as failing for very short test lengths, though all 
that produce results should be able to get close enough to 0.0 to fail for any 
test length that produces a result.  

Tests longer than 64 gigabytes often treat the result distribution as 
identical to that of 64 gigabyte tests, simply to cap the resources needed in 
calibration.  This produces discrepencies too small to be significant for most 
purposes, because the results distribution is generally already fairly close 
to its assymptotic form by that point.  

Do NOT assume that p-values from the same results summary have zero 
correlation.  While the normal test set is *mostly* orthogonal it's not 
*completely* orthogonal.  The significant intra-correlations in the core test 
set are:
* Gap-16:A is correlated with Gap-16:B and Gap-16:C.  The strength of 
correlation is strong for shorter sequences and slowly decreases as sequence 
length increases.  The Gap-16 results should have no significant correlation 
with the other standard tests.  
* BCFN(X1,Y1) should correlate with BCFN(X2,Y2) if and only if X1 and X2 are 
similar - if they differ by 1 then the correlation is moderate, if they differ 
by 2 then the correlation is minor, if they differ by 3 then the correlation 
should be insignificant, and at 4 or more the correlation should generally be 
zero.  The Y values are generally not very relevant.  
* DC6-9x1Bytes-1 is slightly correlated with BCFN(2,whatever) and 
insignificantly correlated with BCFN(3,whatever).  Higher X BCFNs should have 
zero correlation with DC6-9x1Bytes-1.  
* All of the correlations for pairs of subtests described above also exist for 
pairs of subtests that have names identical to the previous described names 
but with matching [LowX/Y] prefixes added on.  
* A subtest with a prefix of the form [LowX/Y] may correlate with 
subtests of the same name but different or no prefixes.  The degree 
of correlation is usually minor among such cases in the core test 
set, though if you have extra foldings enabled it might be moderate 
in some cases.  

(why, you ask, are there moderate or even strong correlations present 
in an optimized selection of tests?  Because the moderate and strong 
correlations are between subtests that enjoy synergies that allow 
them to be more efficiently calculated at once - the efficiency 
bonuses they enjoy thusly are slightly offset by non-orthogonality 
penalties, but only slightly.  The strongest correlation in the core 
tests that is NOT a byproduct of a major efficiency boost is that 
between DC6-9x1Bytes-1 and BCFN(2,whatever), which is only a slight 
correlation)

The expanded test set is less orthagonal than the core test set - many 
of its tests are strongly correlated with each other.  

1.B.8 - Metatests performed on PractRand p-values:
Normal PractRand p-values are usually fine for performing meta-tests 
on, assuming no ~ is shown with them.  However, using large numbers 
of p-values in a single meta-test may be enough to amplify biases in 
the p-values to noticable levels, depending upon the particular type 
of metatest used and how tolerant you are of bias in the result.  
50 p-values should generally be okay, 5 should always be okay, 
50,000 should only be acceptable on a rather limited range of tests 
and metatests (check empirically).  

If RNG_test printed a "~" or (for those linking with PractRand 
directly) TestResult had a type with BAD in the name (eg TYPE_BAD_P), 
or the result type was a pass/fail result, then metatests are 
probably a bad idea.  

Results summaries from the same run of RNG_test are based upon some 
of the same RNG output and thus have some degree of correlation.  
Normally two consecutive results summaries have 50% overlap, producing 
a substantial degree of correlation, but that can vary with the 
command line parameters used.  

1.B.9 - What specific failures might mean about a PRNG

Failures on test names that begin with [Low1/??] (replacing the ?? with 8, 16, 
32, or 64) often mean low-bit issues - not enough right-ward mixing (barrel 
shifts and right-shifts mainly, though also table-lookups and some more 
complex / slower math) in the PRNG.  However, other possibilities aren't rare 
either - it could be that the bit focused on didn't need to be the lowest bit, 
perhaps it just had to be one bit per word to emphasize the relationship at 
that particular spacing.  

Simultaneous failures on multiple unrelated tests often means that the PRNGs 
cycle length was exhausted, or it otherwise changed behaviors to something 
awful near the end of testing.  

P-values near 1 often correspond to cycle exhaustion or subcycle exhaustion 
issues, especially in monocyclic PRNGs.  

There are currently 8 tests in the core test set: one parameterization each of 
BCFN, DC6, Gap-16, FPF, BRank, mod3n, TMFn, and BDayS2.  These 8 are not 
created equal.  The first four of those spend a lot of time and resources 
trying to detect broad categories of biases.  The last four are only alotted 
tiny amounts of CPU time, but they try to catch narrowly defined type of bias 
that are known to slip past the broader tests sometimes.  

Failures on short-ranged tests, especially DC6 parameterizations, are typical 
of small chaotic PRNGs with insufficient mixing / avalanche.  Failures on FPF 
parameterizations can have similar meaning.  However, both of those find bias 
in a wide variety of PRNGs, sometimes for non-obvious reasons.  

BCFN failures where the first parameter listed is high (say, 5 or more) 
typically reflect a cyclic buffer being traversed, where patterns in the 
content of that buffer show up at specific spacings.  On the other hand, BCFN 
failures where the first parameter listed is low are typically similar in 
meaning to a failure on a DC6 parameterization (see the previous paragraph).  
Note that when I refer to the value of the first BCFN parameter, I'm refering 
to the evaluated value listed in the output, meaning that for a test result 
listed under "BCFN(2+3,13-1)" the first parameter would be 5, because "2+3" 
evaluates to 5.  

Failures on BRank suggest that in some way the PRNG output, or at least part 
of it, was extremely linear, producable strictly by xoring bits of previous 
PRNG output.  This is classically seen on LFSRs and equivalents, but sometimes 
it shows up on PRNGs that are *supposed* to be chaotic, but actually have some 
component that isn't really.  

For mod3n failures, where the subtest has a name of the form "mod3n(X):(Y,Z)", 
look at the value of Y there.  If the Y value is near zero, this corresponds 
to a short-range correlation.  If the Y value is high it corresponds to a 
long-range correlation.  This actually looks for simple patterns in the 
value-modulo-3 of the PRNG output.  Most such patterns will be detected by DC6 
or BCFN, but those two tests both look at hamming weights to spot such 
patterns, and once in a while an otherwise very simple bias will show up in 
such a way that is no impact on the distribution of hamming weights.  You 
might ask "why 3?  why not 5 or 17?".  Some biases will slip past mod3n but 
could be caught by a hypothetical mod5n.  However, most biases that could be 
caught by a mod5n but not a mod3n will instead be caught by DC6 or BCFN - the 
biases that can get past DC6, BCFN, and mod3n, but could get caught by mod5n 
are very very very few.  I think.  

TMFn and BDayS2 typically catch particular types of LCGs, but both are also 
occasionally useful against other sorts of PRNGs.  Both look for particular 
kinds of simple long-range patterns.  Despite the similar descriptions, they 
go about things in very different ways and frequently catch different PRNGs.  

Most failures on the core tests (other than BRank) should have started small 
and gotten steadily worse as more bytes were tested until failure was 
reached.  Thus, if "FAIL" was reported at 2**36 bytes then at the previous 
results summary (at 2**35 bytes) there would typically have been some kind of 
anomaly on the same subtest, maybe a "suspicious" or "very suspicious" test 
result.  If there was no preceding warning and the failure was particularly 
extreme, that may mean that the RNG somehow worsened its behavior in some 
way shortly before the failure was reported, similar to exhausting the cycle 
length, but perhaps in some way detectable to only one of the tests in 
PractRand.  

1.B.8 - Commentary on the calibration system used internally.  

Most p-values emitted by PractRand came out of its internal calibration 
system.  This looks at large numbers of results on known good PRNG results to 
establish what the result distribution actually looks like (instead of what 
it's theoretically projected to look like), and then extrapoltes and 
interpolates from that to produce a continuous mapping from test results to 
transformed p-values (transformed so that it's possible to represent how 
extreme they are even when closer to 0 or 1 than floating point values 
normally allow) anywhere from negative infinity to positive infinity.  

It was specifically designed to correct the bias in results of chi-squared 
tests naively used on overlapping samples and then crudely forced to a rough 
approximation of a normal distribution with mean near zero.  It also works 
quite decently for a variety of adjacent cases as well, but it does have a 
number of limitations.  

It has multiple kinds of problems if the underlying distribution is too 
quantized - typically that becomes relevant for very short tests, especially 
when the p-values are near 1.  A number of steps are taken to prevent it from 
causing real problems, and I'm not aware of it ever managing to do so, but 
sooner or later I'll probably want to take more steps.  

It has different issues if there's not enough known good test results to work 
with.  If it has less than 100 results it won't function at all.  At 100 it 
will generally produce moderately accurate results for a central region
0.2 <= p <= 0.8, and increasingly paranoid guesstimations of values the 
farther they are outside of that range.  More samples improves everything.  

There are three primary dangers to having insufficient samples.  The first is 
simple lack of precision - the fewer the samples, the more inherently 
error-prone everything is.  The second is that, because it must estimate 
p-values far outside of the range the samples occupy (partly because PractRand 
establishes failure thresholds in terms of extreme p-values far outside the 
ranges routinely seen), and it has limited information to work with and tries 
to be conservative so as to minimize the possibility of false positives, its 
paranoia may cause failure thresholds to be far more extreme than intended - 
PractRand may try for reject events that occur once in 1e11 times, but 
accidentally end up passing everything up to the level of once in 1e88 times 
instead, requiring significantly more data to reject a bad PRNG than it 
should.  This isn't entirely fixable for the extreme p-values thresholds that 
PractRand likes to look for, but needing to test extra data isn't the worst 
possible outcome, and usually it only takes a weeks worth of dedicating 1 core 
to generating calibration data to make that issue quite minor - if 8 times the 
ideal amount of data needs to be tested, that's a problem worth addressing, 
but if 1.2 times the ideal amount of data needs to be tested then I don't 
really care.  

The last issue resulting from insufficient calibration samples is the most 
serious - if the underlying curves (graph where Y is the log of distance from 
the p-value to extrema, X is vaguely gaussian-ish distribution that 
calibration takes as input) have an inflection point (where the 2nd derivative 
becomes positive) farther out than the p-value corresponding to twice the 
reciprocal of the square root of the number of samples, then it's possible for 
the built-in paranoia to be insufficient, resulting in calibrated p-value more 
extreme than the real p-values should be.  However, unless the underlying test 
made some rather extreme design decisions, this is generally very manageable.  
Simply generating calibration samples for a few weeks is almost always 
sufficient to avoid this circumstance.  Even when calibration data is 
insufficient to avoid this circumstance, it usually is unable to produce 
actual false positives reports - when the bad inflection point is hit, the 
margin of paranoia will start decreasing and may even become negative, but 
generally the final inflection point will be hit long before failure is 
reported, at which point the margin of paranoia will start increasing again, 
and because PractRand sets failure thresholds rather high there will still 
normally be a large buffer between the final inflection point and the failure 
threshold in which the margin of paranoia will recover.  

All that said, there are still potential scenarios in which simply increasing 
the number of samples used for calibration won't help.  If the the input the 
calibration system doesn't broadly resemble the kind of curve I'm expecting, 
then it's possible the final inflection points will have the wrong signs (I 
removed the "all2" subtest of FPF a while ago because I wasn't sure it met 
this criteria).  Alternatively, even if the underlying test does roughly 
correspond to the kind of thing produced by a naively overlapping chi squared 
test, it might not produce usuable results until some minimum length is 
reached - and that minimum could be prohibitively long in some cases, 
rendering the test effectively useless.  In particular, if the base 
probability table before the overlapping window is taken in to account has 
extremely unequal probabilities coupled with a large overlap window, many of 
the buckets in the chi squared test will naturally have low expected values, 
which can be greatly exacerbated by the overlap window, and end up producing 
a distribution that is perfectly usable on assymptotically long data 
sequences but has its variance dominated by extreme outliers on test 
sequences shorter than some threshold, making short sequences unusable.  If 
the degree of overlap is high enough and/or the ratio of probabilities 
between rare and common events pre-overlap is high enough, then the minimum 
usuable length for a test may be prohibitively large, in which case I simply 
won't use the test.  

Anyway, another limitation: because calibration needs large numbers of samples 
from test results on known good PRNGs, that can't realistically be done for 
very long tests.  Usually I do it for sample lengths up to 64 gigabytes, then 
verify by eye that the distribution is sufficiently converged to an 
assymptotic ideal to be usable by vastly longer tests with minimal error, 
extending the data to longer samples if necessary.  Typically the error this 
introduces is on the the level of multiplying the p-value's distance from the 
nearest extrema by a value somewhere between 0.75 and 1.25 on assymptotically 
long test runs (less for values far from extremas) - a big enough error that 
I'd like to reduce it, but not big enough to justify major engineering 
efforts, nor enough to justify spending half a year expanding all the 
calibration datasets enough to cut that error in half.  

And I guess one last limitation is that even below the largest test length 
calibrated, the exact test length used may not exactly correspond to any 
calibration data set.  This is because while PractRand normally only prints 
results for test lengths that are powers of 2, with the right command line 
options it is possible to do test lengths that are any multiple of 1024 
bytes.  There are a finite number of calibration data sets saved, so really 
oddball test lengths might not exactly match any of them.  However, this is 
mostly trivial - most test have sufficient calibration datasets saved that one 
close enough to any test length can be found for the difference to be 
negligible.  If I ever feel particularly paranoid about this I might adjust it 
to try multiple different calibration datasets when an exact match couldn't be 
found and report the least extreme results, but so far this is a non-issue so 
there's not much point.  

Needless to say, I go to considerable lengths to make sure that no scenario 
ever occurs on the core tests that produces false positives where users could 
see them.  And I make some effort for the expanded test set too, though less 
so than I do for the core test set.  If PractRand can't get the false 
positive rate low enough, the the test result is discarded.  If anyone is 
aware of PractRand ever failing a known good PRNG (HC-256, Trivium, ISAAC with 
large table sizes, ChaCha with 8 or more rounds, et cetera) let me know - that 
would be a serious bug.  

TLDR: PractRand p-values are guestimates of varying quality, but you should 
never encounter a false-positive from them.  If you do, it's a bug.

2.A - A Brief historical overview

This is all general theory, not anything specific to PractRand.  

The classic source of empirical statistical tests for random number 
generators is "The Art of Computer Programming", by Donald Knuth.  Several 
empirical tests are described in that book.  

The purpose of such tests is to distinguish data produced by a truly random 
process from data produced by a non-random process.  They look for patterns 
associated with types of not-entirely-random systems that the data might have 
come from instead of coming from a truly random process.  There is a 
theoretical problem here - there are an infinite number of such patterns and 
possible not-entirely-random systems, and if you consider ALL of them then 
all blocks of data are equally likely to be random or non-random.  In 
practice however, this is not actually a problem - most of those infinite 
number of patterns are infinitely complex, utterly impractical to test for, 
and don't really make much sense in the context of math and hardware as we 
know them.  In practice, looking for patterns that tend to arise from non-
random processes of limited complexity and very finite state size built out 
of standard mathematical operations (particularly those that are most 
efficient in binary logic) does a good job.  

In empirical testing, a classic is the Diehard program by Marsaglia.  That 
program is badly outdated today and should not be used anymore, but it was, 
so far as I know, the first real standardized battery of such tests.  A 
standardized battery of tests is just a more clearly defined (and usually 
more convenient) version of running multiple tests.  The basic results of a 
standardized battery of tests can be communicated quickly and effectively 
with just the name and version of the standard battery of tests and the 
length of sequence tested (though some batteries operate on fixed-size 
sequence lengths and don't need the last).  That is much easier to 
interpret (not to mention more concise) than a listing of every major test 
name / version / implementation / parameterization tried.  


***********************************************************************
3. General Information on Interpretting Test Results
***********************************************************************

3.A - Forms of test results

Most such tests produce results of a form called a "p-value".  p-values are 
numbers between 0 and 1 that represent where a tests results fall within the 
range of results expected on truly random data (ie truly random data should 
produce p-values uniformly distributed between 0 and 1).  The results can 
never truly lie outside the range expected from truly random data, but they 
can lie absurdly close to an edge of the expected range.  Tests are generally 
constructed such that the pattern they are looking for produces p-values that 
are very close to zero.  Results that are extremely close to 1 are frequently 
also considered suspicious.  Exactly how close to zero or one might be 
considered a failure varies extremely widely - I have seen sources that 
differed by more than 8 orders of magnitude on the issue.  

Some tests will produce flat pass/fail or pass/ambiguous/fail results in place 
of p-values.  Any pass/fail result should in theory correspond to whether or 
not a hypothetical p-value would be above or below some specific threshold 
(the percent of the time that truly random data would produce a fail result).  
In practice however, it may not always be possible to figure out what 
threshold a given pass/fail criteria corresponds to - in some cases 
theoretically analysis may be intractible while extensive emperical analysis 
might simply determine that the threshold had a 99% chance of being less than 
1e-4.  How much less?  Hard to tell exactly.  

Typically p-value calculations are not 100% accurate (ie not perfectly 
uniformly distributed between 0 and 1 on truly random data).  Instead they 
have small complex deviations from uniformity - though exactly how small 
varies widely.  In some cases these inaccuracies are insignificant, in other 
cases they overwhelm all other factors.  

Similarly a pass/fail result will typically fail a percentage of truly random 
data that differs slightly from the intended percentage.  

A p-value can be converted to a pass/fail result simply by comparing it to an 
arbitrary threshold.  Is P less than 0.001?  Then it failed.  That kind of 
thing.  The exact value of the arbitrary threshold used varies widely 
depending upon who picks it - some prefer thresholds of 0.05, others prefer 
thresholds of 0.0000000001.  This is assuming single-ended failure criteria, 
in which a p-value is only considered bad near zero, not near one.  If the 
p-value has double-ended failures you might have one threshold at 0.0001 
(anything below that is considered a failure) and another threshold at 0.9999 
(anything above that is considered a failure).  

3.B - False positives

The false positive rate is the chance that a perfect RNG would fail a test.  
All meaningful evaluations of RNG quality have a non-zero false positive rate, 
though it can be very very close to zero in some cases.  

In theory, if you're dealing with p-values (under the idealized assumption 
that only values near zero are failures, values near one are not), then the 
false positive rate is equal to the failure threshold.  A failure threshold of 
0.01 means a 1% false positive rate.  

Unfortunately, that does not take in to account inaccuracies in the 
calculations of the p-value, which can have large effects on false-positive 
rates.  Even when no p-value is produced, there is almost always a discrepancy 
between what the false positive rate is predicted to be and what it actually 
is.  

3.C - Practical meaning of false positive rates.  

If you make your conclusions based upon a false positive, then testing was 
worse than worthless.  Generally you thus want something like a 99% chance of 
no false positives in the results your conclusions will be based upon.  

The chance of zero false positives is less than or equal to 1 minus the 
expected total number of false positives.  Thus if your expected total number 
of false positives is 0.1, that means at least a 90% chance of zero false 
positives.  If it's 0.01, that means at least a 99% chance of zero false 
positives, 0.001 would mean at least 99.9% chance of zero false positives, 
et cetera.  

Finding the expected total number of false positives is straightfoward: 
simply add up the false positive rate of every test that could directly 
influence your conclusion.  If you are running 120 tests each with a 0.1%
false positive rate, that means 120 * 0.001 = 0.12, so you have at least 88% 
confidence that no false positive results showed up.  Which isn't quite 
good enough for typical purposes.  

If you're following this then a few questions should occur to you.  Like
these:

Q: What if my expected total number of false positive is greather than 1?  
That would produce a negative chance by that equation, this is obviously 
bogus!  
A: Obviously the actual probability does not go negative.  I did say that was 
a lower bound.  A slightly closer lower bound would be e^-X, where X is the 
total expected number of false positives, so at X=1 the closer lower bound 
says at least a 36.7879% chance of zero false positives.  You could switch 
formulas when X gets big enough, say 0.3 or so.  But frankly, there's no 
point - if X is big enough to switch formulas then your conclusions have a 
high chance of being contaminated by false positives anyway.  

Q: Didn't you just say that false positive rates don't actually match what 
they're supposed to be?  Now we're doing calculations based off of them?
A: Yep.  It's not that bad though, we're adding them up and we only really 
care about the rough log of the sum, so proportionate error on each one is 
fine.  Even if they all have a proportionate error of 100%, that just means 
that what should add up to 0.01 instead adds up to 0.02, giving you a 98% 
chance of zero false positives when the numbers suggest you should have a 99% 
chance instead.  The real problems to watch out for are errors that have 
unbounded scaling with some parameter you might unknowingly mess with, and 
errors in results that then get used as input to metatests.

The expected total number of false positives is equal to the sum of the false 
positive rates of every test you are going to perform.  If the expected number 
of false positives is near zero, then the chance of zero false positives 
occurring is approximately 1 minus the expected number of false positives.  

(It's like a Poisson distribution with lambda equal to the expected total

evaluating the probability mass function at 0, which works out to e^-lambda, 
so if E(total false positives) == 0.01, then the chance of zero false

quite right to do so because they often aren't - but that's no reason to worry 
because the result from treating it as a Poisson is actually a lower bound on 
the chance of zero false positives occurring, because when they clump up, that 
means that other possible result sets must have zero false positives to keep 
the expected number of false positives at lambda.)

Any meaningful scheme for evaluating test results must have a non-zero 
false-positive rate, meaning a chance of incorrectly producing a failure on 
good (ie produced by an ideal random process) data.  If the false-positive 
rate is significantly greater than zero then your test results may not mean 
much on their own.  How much greater than zero is significant depends upon the 
cost of making an error, and how many chances to make an error you will have.  
If you are producing a million test results and want the chance of having any 
errors at all to be on the order of one in a million, then you want a 
false-positive rate on the order of one in a trillion or less.  

A very common approach to dealing with false positive is to run tests 
repeatedly (on different data from the same RNG), and ignore failures that did 
not occur repeatedly, with the number of failures required to be considered 
significant adjusted for the expected false-positive rate.  This is a 
convenient thing to do that can meet peoples' needs, but it's also a horrible 
thing to do that causes a lot of problems.  

How can it be both?  

If you are looking at the results of a statistical test, or even an entire 
battery of statistical tests, and seeing suspicious results, then running it 
again is probably extremely convenient - it's probably just repeating the 
last thing you did.  And it's probably at least slightly useful.  If both 
sets of results are suspicious in almost exactly the same way, well, that says 
something doesn't it?  Not always what you think it says, but it does say 
something.  

But then what happens is someone looks at that approach and says "Aha, this 
must be the correct way of doing things!  More separate runs of the test means 
more information, more sensitivity, and more quality!  I must maximize number 
of test runs, maybe automate this process, produce a p-value for the combined 
result so people can easily assess the combination of many results".  NO.  
Please stop.  Do not do this.  You may be able to reasonably eyeball two or 
three different result sets for suspicious similarities, at least if you have 
some idea what results on good PRNGs typically look like.  But once you start 
systematically combining larger numbers of test results in to a single 
pass/fail criteria or a single p-value, things tend to go wrong.  Combining 
results REDUCES the quality of the result.  By which I mean that combining 
results reduces the amount of information you know about the false positive 
rate, and/or it reduces the uniformity of the p-value distribution.  Combine 
enough results naively and eventually you'll either fail every possible PRNG 
or pass every possible PRNG - this is not just hypothetical, I run in to 
these kinds of issues in popular software while attempting to follow the 
users manual.  Such issues can be addressed to an extent, but doing so is 
usually not worth the effort for normal users of statistical tests, and each 
solution will have major problems its own - very limited ranges of numbers of 
samples they work adequately for, only working for one specific test or 
narrow type of test, extremely poor performance, that kind of thing.  
Combining results *does* improve sensitivity, but it is an inefficient way to 
do so - typically, if you test 1 gigabyte of PRNG output 100 times and 
combine the results, you'll get about the same level of sensitvity as if you 
had ran the test on 10 gigabytes once.  So unless for some you *can't* run the 
test longer (for instance, if the test is hardwired to once specific length), 
it's better even from a sensitivity standpoint to run the test longer than to 
repeat the test.  

Anyway.  Another aproach is to set the false-positive rate to approximately 
zero.  Of course, you can't set it to exactly zero, but if the false positive 
rate is less than 1 in a million, well, that's good enough already for many 
uses, and it's low enough that uses that require even lower false positive 
rates can easily achieve without having to combine more than two or three test 
results, minimizing issues caused by combining test results.  


3.D - Metatests

Sometimes instead of considering a tests p-values as a final product 
someone will use it as random data to be put in to another set of 
statistical tests.  Such a test using other tests p-values as input 
is referred to as a metatest.  

P-values for data from good RNGs are supposed to be uniformly 
distributed between 0 and 1.  Test results on data produced by 
different seeds should show no correlation.  Thus p-values from a good 
test run on different seedings of a good RNG should behave like the 
output of a good RNG.  Care must be taken to avoid using correlated (or 
at least strongly correlated) test results in such a way, but this is 
generally a managable problem for a canny user.  

Some test suites have a habit of completely hiding p-values produced 
directly by their basic statistical tests on the random numbers and 
displaying instead metatest results.  

But why would such a thing be desirable?  Here are a few reasons, 
and some associate drawbacks for each:
* If the original test results p-values have biases with known 
properties, this can be an effective way to remove their biases.  
- However this can amplify errors if done incorrectly, and is 
not necessarily the most efficient way to go about correcting 
well understood p-value biases.  
* By combining multiple tests, the sensitivity of a test can be 
scaled up.  For some tests, this may be the *only* practical way 
to scale up their sensitivity.  
- However, in practice this is a very poor way of scaling up a 
tests, so poor that in all cases I have examined you would be 
at least as well off leaving the test unscaled or abandoning the 
test entirely.  
* If there are multiple tests and we don't want to look at their 
individual results but a combined result instead.  
- I have found using extremely crude metatests to be more useful 
for this purpose than sophisticated metatests due to the costs 
imposed by having to eliminate all correlation between the 
various base tests (sophisticated metatests generally require 
that all inter- p-value correlations be completely understood, 
which is generally only possible if all such correlations are 
indistinguishable from zero).  

(PractRand does not encourage the use of metatests by end 
users; metatests are used in the validation and calibration 
of PractRand subtests though.)
