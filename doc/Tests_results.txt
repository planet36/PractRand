This is about how I test the tests themselves.  I have used this methodology 
for both individual tests and entire batteries of test.  Basically, I have a 
very large set of bad PRNGs.  Deliberately bad - if I think a PRNG is too 
good to be rejected by any modern (or near future) tests then I'll discard 
it.  Then I use semi-automated means of testing the entire set with various 
different test batteries or individual tests and collating the results.  
This helps me figure out which tests are worth using, which aren't worth 
using, and what sorts of tests should or shouldn't be added to a battery of 
tests.  

You might find the sets of PRNGs and scripts useful if you are trying to 
test statistical tests.  Alternatively, you might find the results of 
various tests summarized here useful in other ways.  This can answer broad 
questions like "which test batteries are best?" (answer: PractRand of course, 
though back when I first started these charts gjrand outperformed PractRand 
version 0.92.  But since then I've updated PractRand several times, 
with making it outperform other test batteries on these kinds of metrics as 
one of my main goals).  Or it can answer narrower questions like, "what PRNG 
algorithms pass TestU01 *Crush but fail Dieharder?" 
(answer: simpleA, binarymult32, mulcrx64, ranrot32, and genindF(2)... plus a 
lot more if you count those that pass BigCrush but fail Crush, which is 
surprisingly common).

PractRand std:
	The software this document accompanies.  
	configuration / usage:
		The defeault settings on PractRand - core tests with standard folding.  
		Used in single-threaded mode (also the default setting).  
	speed:
		On my computer, this tests about 256 gigabytes per hour on fast PRNGs.  
		Time used is mostly linear with amount of data tested... 
			not counting the time taken to generate interim results, 
			which is usually between 0 and 2 seconds per set of results shown.  
	strengths:
		Exceptionally effective against a wide variety of types of PRNGs.  
		Uses the least CPU time per biased PRNG found.  
		No (realistic) maximum test length.  
		Command line interface is easy to script with.  
		...and also can be linked with and used as a C++ library.  
		Output format is easy to parse, both for humans and scripts.  
		Best overall results.  
			Of course, to some extent that's a function of which PRNGs are tested
			which I picked out
		The only test suite to offer interim results.  
	weaknesses:
		Cheats slightly on all RNGs by looking at the metadata - 
			it treats 8 bit RNGs slightly differently than 32 bit RNGs, etc
		Tests are not completely independent.  
		Requires lots of random numbers.  
			if the PRNG being tested is very slow, this could be a problem
	record notation:
		Test failures are recorded as the smallest data size at which failure was detected.  
			only sizes that are powers of 2 are tested, starting from 1 kilobyte
		If no failures were found, instead the result is recorded as > X
			where X is the largest size tested.
		Failing is anything that PractRand explicitly evaluates as FAIL.  
			Anything short of that is considered passing.  
	link: https://pracrand.sourceforge.net/

gjrand pmcp:
	Quite obscure, but actually really good.  
	configuration / usage:
		I use pmcp (I think that stands for Parallel Master Control Program?)
			I think mcp is equivalent, but doesn't share PRNG output between tests so it's slower.  
		with the command line option for one of its standard sizes:
			--small, --standard, --big, --huge, --tera, or --ten-tera
		It also supports --tiny, but that's not much faster than --small so I don't bother with it.  
		WARNING: only run one instance of pmcp at once.  Otherwise it can SILENTLY FAIL.  
			Meaning it may produces output that superficially LOOKS correct but is actually GARBAGE.  
		WARNING: the present working directory must be the path where pcmp and the other executables are located
		I'm using gjrand version 4.2.1.  
			There also exists a version 4.3.0, but I rejected it in favor of 4.2.1
			Why?  ...I don't remember.  Might have been important, or maybe it was trivial.  
	speed:
		On my computer, --small takes about 10 seconds when using a single core
			--standard takes about a minute
			--big takes about 5 minutes
			--huge takes about an hour
			--tera takes about 8 hours
			--ten-tera takes about 4 days
		There's also --tiny, which takes about 3.5 seconds, but I don't use that one much.  
	strengths:
		Highly effective against a wide variety of types of PRNGs.  
		Second best overall results by the metrics these charts focus on, after PractRand.  
		Can useful summarize results with a small number of p-values.  
		Number of failures found is more meaningful than with most other test batteries
			In other batteries a single test algorithm rejecting a PRNG might produce anywhere from 1 to 1000 failing p-values (and thus that many reported failures), depending upon how that test was structured.
			With gjrand, every testing algorithm produces exactly 1 p-value, and gets reported as exactly 1 failure.  Thus the maximum possible number of failures is only about 13.  
		Decent command line interface aside from the problems mentioned earlier.  
	weaknesses:
		See the all-caps warnings in the configuration/usage section?  Those matter.  
			Especially the first one.  It's incredibly painful if you have a large archive of test results, and then you realize that some of them are garbage but you can't even reliably tell which ones, and it will take months to rerun all your tests...  
			Even aside from the problems with garbage output, lack of being able to run multiple instances at once can be a problem for some users.  Like me.  
		I haven't dared to even try to build this on windows.  I doubt it would go well.  
	record notation:
		The results are recorded as a sequence of up to 6 characters.  
			The first character represents the result of pmcp --small, then --standard, --big, --huge, --tera, and --ten-tera, in order.  
			If no failures were detected, that is recorded as dash character (-) or a tilde character (~)
				depending upon whether or not suspcious but non-failing results were present.  
				for this purpose, I consider a test result as a failure if pmcp reports it as a grade 10 or above failure
				and I consider a test result suspicious if pmcp reports it as grade 4 through 9 failure.  
			Otherwise, the character is the number of tests failed, in hexadecimal, which can range from 1 to D.  
			Thus, the best possible result is ------ (every test result was reported as no worse than grade 3)
			The worst possible result is DDDDDD (every test result was reported as grade 10 or above)
			If the result is shorter than 6 characters that means that the longest tests haven't yet been run.  
				Often longer slower tests are omitted for lack of time, especially if shorter tests already showed decisive results.  
	link: https://gjrand.sourceforge.net/

TestU01 *Crush:
	Meaning SmallCrush, Crush, and BigCrush from TestU01.  
	configuration / usage:
		Just the normal behavior of SmallCrush/Crush/BigCrush.  
		The only thing I've meaningfully adjusted is interface.  
	speed:
		On my computer, SmallCrush takes maybe 10 seconds on very fast PRNGs.  
		Crush, about half an hour, and BigCrush takes maybe 4 hours.  
	strengths:
		Can be linked with and used as a C++ library.  
		Good academic credentials behind this.  
			And it shows - you can read the paper this was published with, for instance.  
		Decent results on most categories of PRNGs.  
		Third best overall results by the metrics these charts focus on, after PractRand and gjrand.  
		Guarantees that all subtest results are independant.  
		Decent output summary format.  
	weaknesses:
		It actually does not test the lowest bit of PRNG output.  
			Which is ridiculous.  It got them some advantages, but... it's still ridiculous.
		I had to write my own scriptable interface (testing piped input etc).  
		I have TestU01 working on Windows, but many have severe trouble with that.  
		Because PRNG output is not shared between subtests, 
			this is more sensitive to PRNG speed than you might expect.  
		Doesn't support multithreading.  
		Contrary to what you might expect, it's common for a PRNG to fail Crush yet pass BigCrush.  
			So, you should never test with *only* BigCrush.  
	record notation:
		If no failures were found the result is recorded as pass if there were no suspicious results, 
			or ~pass if there were suspicious results.  
		Otherwise, results are recorded as X/Y/Z
			Where X, Y, and Z are decimal numbers of tests failed.  
			X is the number of tests failed in SmallCrush, Y corresponds to Crush, and Z to BigCrush
			X, Y, or Z may also be ~0, which means no failure but at least one suspicious result.  
		Thus, a result of 0/1/2 means SmallCrush was passed, but 1 subtest of Crush and 2 subtests of BigCrush were failed.  
		Any p-value less than 1e-9 or greater than (1-1e-9) is considered a failure.  
		Any non-failing p-value less than 1e-4 or greater than (1-1e-4) is considered suspicious.  
	link: https://simul.iro.umontreal.ca/testu01/tu01.html

RaBiGeTe:
	I'm now using version 2.0 of this.  
	configuration / usage:
		I use a custom configuration, with 4 tests active: Short block, Windowed Autocorrelation, Rank, and Permutation
			In my testing, Short block was by far the most important test, followed by a roughly 3-way tie between the other 3.  
			Some other tests showed some value, but in the vast majority of cases it seemed that giving more CPU time to those 4 tests produced better outcomes than enabling a wider variety of tests.  
			For Short block and Windowed Autocorrelation, I used the parameters from the Good'n'Fast preset.  
				This was important, because using the parameters from the Extended preset resulted in too many p-values, causing the interface to become effectively unusable on longer test runs.  
					Seriously, the program would lock up for multiple hours any time I asked it to do something simple like sort the p-values.  
			For Rank and Permutation, I used the parameters from the Extended preset.  
		If you wish to run a wider variety of test, my suggestion is start with the above 4, then add: Runs, Weight, Block gap dist, OET, Peak, Coupon, and the various Random Walk tests.  
			That excludes the Partition, Linear Complexity, Serial, Non overlapping template matching, and DFT tests.  
				Those tests take a ton of CPU time, but rarely detected any biases, and I've never seen them detect any that hadn't already been detected by other tests using the same or less data.  
			That also excludes AMLS, Collision, Maurer, Long block, Blocks'n'Gaps, Serial two-bit, and Cumulative sums tests.  They don't take much CPU time, but contributed basically nothing.  
		I test only one sample.  I then sort the p-values for that sample.  If any p-value is "fail" or less than 1e-10, that is counted as a failure.  
			Note that this is contrary to the recommendations of the author, but it gave me better results than other methods I tried.  
			p-values close to 1.0 are not considered failures - it frequently generates those even on known good PRNGs.  
	speed:
		I ran tests of lengths 1 megabit, 8 megabits, 32 megabits, 128 megabits, and 1 gigabit.  When I have more time I'll add 4 gigabits as well.  
			Note that those are in units of *bits* not bytes, unlike for other test suites.  
		Those test lengths took about ~2 seconds, ~9 seconds, ~50 seconds, ~4 minutes, and ~45 minutes, when forced to run on a single core.  
			4 Gb looks to take a touch under 4 hours.  
		If I have more time I might run longer tests on some PRNGs.  
	strengths:
		RaBiGeTe does not need as many bits to detect bias as most other test batteries.  
		It performs quite decently on several categories of PRNGs.  
			Not as wide a variety as the above test suites, but still a wide variety, and better than the above on some within that variety.  
		Includes a GUI that lets you examine and visualize test results in a variety of ways.  
		You can link in your PRNGs object code via dll.  
	weaknesses:
		It's fairly slow on a per-bit-tested basis.  
		Scripting and automation of tests with this seems pretty much impossible, given the GUI interface.  
			In particular, you can't pipe PRNG output to it.  PRNG's must either be written as a dll, or have output saved to a file.
		The maximum test length seems to be just 4 gigabits.  You can set it to test more, but it silently adjusts test length down to 4 Gb.  
			That's about a 3 hour test on my computer, btw.  
		There are several categories of PRNGs it's hopeless against.  
			For instance, large PRNGs.  Everything has trouble with them, but RaBiGeTe suffers even more than other test batteries when PRNGs get big.  
				An example is mm32 (Mitchell & Moore's lagged sum PRNG, an old medium-large PRNG of very poor quality).  
					EVERY other test battery (except STS, which shouldn't count) easily finds bias in mm32.  
		Windows-only and closed source.  
		It supports multithreading, but... actual performance is not matching its settings well in the current version.  
			It sometimes maxes out all cores even when I tell it to only use 1 thread.  I've taken to locking it to a single core via CPU affinities in the task manager to avoid this.  
			I'm not seeing the speedup I expect when it uses multiple cores.  I remember it getting more speedup from cores in an earlier version, but maybe my memory is wrong.  
		Crashes once in a rare while.  
	record notation:
	link: http://cristianopi.altervista.org/RaBiGeTe_MT/

Dieharder:
	I'm using version 3.31.1
	configuration / usage:
		I use 4 different sets of tests within dieharder, which I refer to as DH0, DH1, DH2, and DHA.  
		DHA involves invoking dieharder with the -a command line option, for all tests enabled.  
			I do not recommend this, I just include it for completeness.  
			The *reason* I don't recommend this is that several tests have serious problems.  
				And many tests show little or no ability to detect bias.  
				And the underlying framework limits the range of parameters to tests too much.  
			ONLY the following tests show significant ability to detect bias:
				dab_monobit2, dab_dct, dab_filltree, dab_filltree2, diehard_opso, diehard_oqso, dab_bytdistrib
			Tests with particular problems (at default parameters) include:
				MAJOR p-value issues
					these issues result in false positive results occuring at significant rates
					specific tests with this issue include:
						marsaglia_tsang_gcd, sts_serial, rgb_lagged_sum
				minor p-value issues
					these issues rarely result in a good PRNG producing a fail message
					but if used with meta-tests that amplify p-value issues even a little...
						then problems become likely
					specific tests with this issue include:
						diehard_sums, diehard_runs, rgb_kstest
				performance issues
					these tests take a long time
					specific tests with this issue include:
						marsaglia_tsang_gcd, diehard_dna, rgb_bitdist, rgb_minimum_distance
				data-dependent lock-ups
					these tests sometimes just grind to a halt, depending upon the data being tested
					specific tests with this issue include: diehard_2dsphere
		DH0, DH1, and DH2 use the following tests
			_     dab_monobit2  dab_dct  dab_filltree2  dab_filltree  dab_bytedistrib
			DH0:  100000000     100000   --             --            --
			DH1:  500000000     500000   5000000        15000000      51200000
			DH2:  2000000000    3000000  48000000       48000000      512000000
			The numbers listed are those used with the -t command line option.  
				Where no number was listed, that test was omitted entirely.  
			Thus, the actual command lines for testing ranrot32 with DH0 look like:
				PractRand/RNG_output ranrot32 inf 0 | dieharder -g 200 -D 504 -d dab_monobit2 -t 100000000
				PractRand/RNG_output ranrot32 inf 0 | dieharder -g 200 -D 504 -d dab_dct -t 100000
			To go over the full details:
				The calls to PractRand/RNG_output is to get random bytes to test.  
					the "ranrot32" argument is the name of the generator to use
					the "inf" argument is the number of bytes to generate, infinite in this case.  
					the "0" is the seed to use
				The calls to dieharder are to run the tests
					the "-g 200" argument specifies that the bytes to test will be piped in throuh stdin
					the "-D 504" argument specifies the formatting that the results will be printed with
					the "-d dab_monobit2" or "-d dab_dct" arguments specify which test to use
					the "-t 100000000" or "-t 100000" arguments specify how many bytes to test
	speed: DH0 takes about 10 seconds, DH1 takes about a minute, DH2 takes about 5 minutes, and DHA 
		takes about 1 hour.  
	strengths:
		The command line interface is decent for scripting.  
		The output format is fairly readable.  
	weaknesses:
		I did mention that some tests have serious false positive issues.  
		And others have milder p-value problems.  
		And some lock up on some data.  
		And most aren't any good at finding bias.  
	record notation: in the "others" column I listed DH0, DH1, or DH2, 
		whichever is the smallest test to fail a particular PRNG.  I 
		list DHA seperately in the same column if it fails that.  
	link: https://webhome.phy.duke.edu/~rgb/General/dieharder.php

NIST STS (Statistical Test Suite)
	I'm using a heavily customized version of STS 2.1.2.  My variant 
	includes various alterations to improve scriptability, speed, and 
	memory requirements.  
	configuration / usage:
		I've adjusted ALPHA from 0.01 to 0.001, the lowest value within 
		their recommended range.  Aside from that, these are the 
		parameters I'm using:
		name      sequence length    num of sequences   disabled tests
		STS0      256 kilobits       32                 TEST_FFT
		STS1      2 megabits         32                 --
		STS2      16 megabits        32                 --
		STS3      128 megabits       32                 TEST_OVERLAPPING
		Why?  Well, first of all, if you're using the results shown in the 
		finalAnalysisReport, which are all basically metatests, then you 
		can't reduce number of sequences much below 32 or the metatests 
		it shows will be worthless.  Secondly, if your sequence lengths 
		are too long then the overlapping templates test will produce 
		false positives.  Likewise, if your sequence lengths are less then 
		about 1 megabit then the FFT test will have problems, and if your 
		sequence lengths are less than half a megabit then the universal 
		test will have 	problems.  Also, if you see the random excursions 
		text or its variant produce large numbers of slightly suspicious 
		rows all at once, it's probably not important - I've seen it do 
		that on multiple generators, and in every case so far it has been 
		non-replicable when longer tests and/or different seeds were 
		tried.  
		Asside from those issues, more time testing is better, but... 
		not by much.  A few seconds of gjrand testing outperforms an hour 
		of STS testing by a very large margin.  Longer sequences generally 
		work better than more samples, for a given total quantity of bits 
		being tested.  When I have more spare CPU time I might add an STS4 
		with longer (1 Gb) sequences, but there wouldn't be much 
		difference anyway.  Plus I'd have to disable several tests - FFT, 
		overlapping templates, and longest runs all yield garbage results 
		at that test length.  
		Anyway... as for what I consider a failure or suspicious...
		since I'm working with finalAnalysisReport as-is, I'm limited 
		to the two metatests it emphasizes.  Unfortunately, both tend to 
		miss failures present in underlying data and exaggerate biases 
		in the underlying tests, but they can be worked with.  The 
		fractions shown on the right-hand side are more useful than 
		the distribution on the left (the p-value summarizes the shown 
		distribution only, that's all it does) - for those, for any 
		tests *other* than the random excursions & variants, any 
		result less than or equal to 29/32 is suspicious, and any 
		result less than or equal to 27/32 is a failure.  For the 
		random excursions & variants things are a bit messier, but 
		any result where the numerator is at least 3 less than the 
		denominator is suspicious, and any where it is at least 5 less 
		than the denominator is a failure.  MAKE SURE you have the 
		value of ALPHA adjusted to 0.001, otherwise very different 
		thresholds should be used (24/32 for failure and 27/32 
		for suspicioun on the default ALPHA of 0.01).  
		As for the metatest shown on the left with the histogram and 
		p-value... I count it as suspicious if the p-value is less 
		than 0.001 (the same threshold it's marking with an asterisk 
		if you have adjusted ALPHA the way I suggested) AND the 
		histogram entry with the highest number (or at least one tied 
		for highest) must be either the leftmost or rightmost column 
		AND the test is not overlapping templates.  I count it as a 
		failure if it meets the requirements to be suspicious and 
		either the two left-most columns combined or the two 
		rightmost colums combined contain at least 16 samples (out of 
		32 total).  
		Also, I completely ignore the left-hand metatest for the 
		OverlappingTemplate test on STS2, and the other other 
		metatest for that test on STS3 and longer.  Quality on that 
		one degrades as sequence length (multiplied by square root 
		of number of sequences) increases, and by STS2 it's marginal, 
		in STS3 and longer it's too broken to safely use.  
	speed:
		Slow.  However, it's difficult to talk with precision on the 
		subject, because numerous different parties, myself included, 
		have applied different custom optimizations to STS.  Still, 
		I'm counting STS0 as taking 10 seconds, STS1 as taking 1 minute, 
		STS2 as taking half an hour, and STS3 as taking	an hour.  
	strengths:
		Name recognition.  This was published by NIST.  
			National Institute of Standards and Technology
				under the U.S. Department of Commerce.
			and thus gets pushed hard by influential parties,
				meaning legislators, regulators, and Pointy Haired Bosses.
	weaknesses:
		Very small variety of biases detectable.  
			Generally not worth testing with unless it's mandatory.  
		Very slow performance on a per-bit-tested basis.  
			Pretty much everyone who uses this more than a tiny bit ends up 
				customizing it to improve speed.  
		Very high memory requirements.  
			Pretty much everyone who uses this more than a tiny bit ends up 
				customizing it to improve memory usage.  
		Very poor output format
			Summary result shows metatests only
				which completely ignore the degree of failure on the root tests
				where the best data is
	record notation: in the "others" column I listed STS0, STS1, STS2, or STS3, 
		whichever is the smallest test to fail a particular PRNG.  In rare 
		cases I prefix the result with a tilde to show that it's more a 
		suspicious result than a failure, but it's on the edge of a true 
		failure, with a different seed it might fail.  
	link: https://csrc.nist.gov/projects/random-bit-generation/documentation-and-software

	notes on the meaning of *lcg names:
		lcg(X,Y) means an LCG with modulus 2^Y, discarding (Y-X) low bits to produce X bit output
		clcg(X,Y) means lcg(X,Y-32) combined with a 32 bit non-power-of-2-modulus LCG
		xlcg(X,Y) is similar to lcg(X,Y), but xor is used in place of addition
			(i.e. equivalent to suppresing the carries on the addition in binary)
		cxlcg(X,Y) means xlcg(X,Y-32) combined with a 32 bit non-power-of-2-modulus LCG
		bigbadlcg(X,Y,Z) aka bblcg(X,Y,Z) means an LCG with modulus 2^Y, discarding (Y-X) low 
			bits to produce X bit output, with a low-quality multiplier of (1 + 2^Z)

	note: I wrote a few generic indirection-based PRNGs so there's more variety here.
	They're generally pretty mediocre.  To keep quality low enough to find flaws 
	in I mostly kept their accumulator / mixing pool down to a single word, and 
	limited their word size to 16 bits.  To determine the size of their indirection 
	arrays I gave them each a parameter that is the log2 of that size, and started 
	it at the lowest value to fail at least 3 different test batteries (usually 
	PractRand and gjrand make up two of those three, and the third is either 
	TestU01 or Dieharder).  If that resulted in a size too small for indirection 
	to occur then the PRNG was discarded.  From there I increased the 
	log2-of-array-size parameter by 50% of the lowest value of that parameter, 
	rounded down to a minimum of 1, and kept increasing it by that amount until it 
	looked likely to pass all test batteries at the next size.  


*****************************************************************************
3. Details of the results format
*****************************************************************************

columns:
	RNG
		The name of the RNG algorithm used;
		The name used is the name it is called in PractRand, not necessarily 
		its real full name - for instance, hc256 instead of HC-256.  
	PractRand standard
		The PractRand standard tests are those returned by 
		PractRand::Tests::Batteries::get_standard_tests().  
		quirks:
			test set is (slightly) dependent upon RNG metadata
		speed on a fast RNG on my computer (in single-threaded mode):
			1 GB: ~15 seconds
			1 TB: ~4 hours
		format:
			The length of sequence required for the PractRand standard tests 
			to detect bias.  If multiple seeds were tested, and bias was 
			found in them at different sequence lengths, then the median 
			result is used.  If bias was not found then the length of the 
			longest sequence tested is recorded, with a ">" prefixed in 
			front of it.  
			Bias is considered to have been found when the standard tool 
			rates the result as "FAIL" (with or without exclamation marks).  
			(p-value threshold vary with circumstances but are typically 
			near 1e-10 on the low side and 1 - 1e-10 on the high side)
			The length is recorded as a number followed by KB, MB, GB, or 
			TB.  Those stand for kilobyte, megabyte, gigabyte, or terabyte 
			respectively.  These are the binary size prefixes, powers of 
			1024, not the decimal size prefixes that are powers of 1000 - 
			while standards bodies advocate weirder names for powers of 
			1024, I follow the de facto standard of powers of 1024 when 
			dealing with units of binary information (including 
			derivatives of those, like binary information per time) and 
			powers of 1000 when dealing with any other units.  
			All lengths are rounded up to the nearest power of 2.  
	TestU01 *Crush
		This refers to the batteries "SmallCrush", "Crush", and "BigCrush" 
		in TestU01.
		quirks:
			does not accept a sequence length parameter
			works on RNGs outputing 31 bits per call:
				on 32 bit RNGs it works but disregards the lowest bit?!?
				on 8/16 bit RNGs, I concatonate multiple outputs together to produce a 32 bit output
				on 64 bit RNGs... I am currently discarding the upper 32 bits to produce a 32 bit output
					might, in the future, change that to returning half and buffering half for the next call
			does not print any results until all are ready
		optimal speed on my computer:
			SmallCrush: ~8 seconds
			Crush:      ~30 minutes
			BigCrush:   ~4 hours
		format:
			If all 3 are passed then the result is recorded as "pass".  
			If all 3 are passed, but only barely, the result is recorded as "~pass".
			If all 3 are passed, but it is noticed that bit-reversed versions of the PRNG fail one or more, 
				the result is recorded as "*pass" (note that this case is not checked systematically)
			Otherwise, the result is recorded as X/Y/Z, where X is the 
			number of tests failed in SmallCrush, Y is the number failed in 
			Crush, and Z is the number failed in BigCrush.  
			In some cases, Y or Z might be recorded as "??" meaning that the correspond test was not run - this is generally only the case if the preceding test failed large numbers of subtests (5 or more failures in SmallCrush, or 10 more failures in Crush).
			Failure is defined as a p-value <= 1.0e-10 or >= (1 - 1.0e-10)
	RaBiGeTe extended
		This refers to the preset testing parameters named "extended", but see the quirks section.  
		quirks:
			In order to get 512 Mb samples to run on extended I had to disable the DFT test
			I used a more recent version of RaBiGeTe that has not been publicly released at this time.  
			I used RaBiGeTe in a slightly non-standard way, testing single long samples instead of many short samples.  
				In my testing this method worked much better than the recommended way.  
			It crashed several times during testing.  
		speed on a fast RNG on my computer:
			16 Mb: just over 1 minute
			512 Mb: a little under 40 minutes
		format:
			The same format was used as for PractRand standard, with three exceptions.  
			A result was considered a failure if any p-value was <= 1e-10, or if 5 or 
			more p-values were <= 0.000001 or >= 0.999999, or if any p-values were 
			omitted from the results.  I never consider a single p-value close to one  
			a failure in RaBiGeTe even if it is exactly 1.0.  
			The second exception is that lengths are listed in bits instead of bytes.  
			The third exception is that if an RNG passed 512 megabits of testing then it was listed as "pass".  
				... because RaBiGeTe was unable to test sequences longer than 512 Mb on the settings I used.  

	If a result is listed as "?" that means that that test has not been run 
	on that RNG.  If a result has a normal value followed by a "?" that 
	means that either that result is strongly implied by tests on nearly 
	equivalent RNGs, or that the test was run and is believed to have 
	produced that result but the record-keeping involved was incomplete and 
	I can't be sure without re-running the test.  


*****************************************************************************
4. Tests that were not included
*****************************************************************************

tests that were not included:
	PractRand expanded standard
		This test battery was not used because:
			A. It significantly underperforms PractRand standard on fast RNGs.  
			B. If I listed its test results here I'd have to update them 
				every time I added a new test to it.  
	TestU01 Rabbit
		This test battery was not used because:
			A. It consistently crashes if used on a long sequence
			B. Its time to complete a test does not increase linearly with 
				sequence length - in some cases it actually decreases with 
				length.  This appears to be partially caused by the test 
				set being a function of the testing length.  
			C. Its results do not seem to improve in power with sequence 
				length consistently enough.  
	TestU01 Pseudo-diehard
		This test battery was not used because its quality is too low.  
	NIST RNG tests
		This test battery was not used because:
			A. Severe difficulty with longer test runs due to not only 
				keeping the whole sequence in memory, but also 
				inflating the sequence to store only one bit per byte.  
			B. Poor quality relative to speed, poor maximum quality 
				given the short maximum sample lengths.
